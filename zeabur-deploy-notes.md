# Zeabur 部署与排错说明

这份文档只记录当前项目在 Zeabur 上已经踩过、且后面最容易重复忘记的问题。

适用项目：`acg-faka`

## 1. 当前部署结论

- 这套项目可以在 Zeabur 上稳定运行。
- 目前遇到的问题，主要来自“传统 PHP 项目 + 容器平台”的适配，而不是业务逻辑本身不可用。
- 一旦镜像、扩展、挂载、配置固定下来，后面可以正常运营。

## 2. 环境前提

- 技术栈：`PHP + MySQL + Redis`
- 数据库名：`zeabur`
- 数据库前缀：`acg_`
- 数据库主机：`mysql`
- 数据库账号：优先使用 `acgapp`

说明：

- 不建议依赖 `root` 作为应用连接账号，Zeabur 环境下可能出现权限问题。

## 3. 已确认的关键修复

### 3.1 挂载卷会覆盖项目目录

现象：

- 首次部署后首页或后台出现 `404`
- `config`、主题、安装文件被空卷覆盖

处理：

- 仓库中已经通过 `Dockerfile` 和 `docker/entrypoint.sh` 增加了默认文件补齐逻辑
- 即使挂载卷是空的，也会自动补齐默认配置、主题和安装资源

结论：

- 后续如果重新部署，仍要优先怀疑“卷把目录盖空了”

### 3.2 安装锁会造成假安装状态

现象：

- 数据库表为空
- 但系统判断为已安装
- 首页或后台直接异常/404

处理：

- 重装前删除：

```text
kernel/Install/Lock
```

结论：

- 如果数据库空了，但系统还显示已安装，先查 `Lock` 文件

### 3.3 Session 必须使用 files

现象：

- 后台登录验证码一直错误
- 验证码明明对，但提交失败

原因：

- Zeabur 环境下 Redis Session 表现不稳定，导致验证码校验时 Session 丢失

处理：

- `php.ini` 中已切换为：

```ini
session.save_handler = files
```

结论：

- 当前项目在 Zeabur 上优先使用文件 Session，不要随便改回 Redis

### 3.4 共享货源导入依赖 bcmath

现象：

- 接入货源时报错：

```text
Call to undefined function Kernel\Util\bcadd()
```

原因：

- 镜像中缺少 PHP `bcmath` 扩展

处理：

- `Dockerfile` 已补上 `bcmath`

结论：

- 后续如果重新换基础镜像，必须确认 `bcmath` 还在

### 3.5 共享货源导入曾经吞异常

旧现象：

- 点“保存”时只出现红叉或空白提示
- 不知道具体哪个商品失败

处理：

- 已修复后台导入逻辑
- 现在会直接返回：
  - 失败的商品名
  - 失败原因

结论：

- 后续如果导入失败，先看页面返回的具体错误，不要再盲猜

### 3.6 远端商品无封面时也可能导入失败

现象：

- 开启“远端图片本地化”时，某些商品导入失败

原因：

- 某些远端商品没有 `cover`

处理：

- 已增加兜底逻辑，无封面时使用：

```text
/favicon.ico
```

## 4. 接入货源时的使用规则

### 4.1 首次测试的推荐方式

先只测最小闭环，不要一上来全开：

- 只导入 `1` 个商品
- `加价模式` 先选 `普通金额加价`
- 加价先填 `1` 或 `2`
- 先关闭：
  - `远端图片本地化`
  - `远端信息同步`
  - `远端价格同步`
  - `远端配置同步`
  - `立即上架`

### 4.2 百分比加价的填写规则

这里不是填整数百分比，而是填小数：

- `10%` 填 `0.1`
- `20%` 填 `0.2`
- 不要填 `10`、`20`

否则价格会被放大很多倍。

## 5. 每次部署后的检查顺序

每次 `Redeploy` 后建议按下面顺序验收：

1. 首页能打开
2. 后台能登录
3. 验证码正常
4. 应用商店可进入
5. 共享货源能拉出远程商品
6. 单个商品可导入
7. 商品页可展示
8. 下单流程正常
9. 支付回调正常
10. 发货正常

## 6. 出问题时优先检查什么

### 6.1 如果是首页/后台 404

优先检查：

- `config` 是否被挂载卷覆盖
- `kernel/Install/Lock` 是否导致假安装
- 数据库表是否真的存在

### 6.2 如果是验证码/登录异常

优先检查：

- Session 是否还是 `files`
- `runtime` 目录是否可写

### 6.3 如果是接入货源失败

优先检查：

- 页面上返回的具体商品错误
- 远端商品数据是否异常
- `bcmath` 是否已安装进当前镜像
- 是否只导入了一个商品做测试

### 6.4 如果是重新部署后行为变了

优先检查：

- Zeabur 是否真的重新构建镜像，而不是只重启
- 挂载卷内容是否覆盖了新代码
- 当前镜像是否已经包含最近的 Git 提交

## 7. 运营建议

- 继续使用 Zeabur 没问题，不需要因为首轮部署踩坑就推翻重来
- 当前最重要的是把“最小可运营链路”固定住：
  - 商品展示
  - 下单
  - 支付
  - 发货
- 插件、主题、企业版、复杂货源同步，建议放在核心链路稳定之后再继续加

## 8. 后续如果继续出问题

排查顺序固定为：

1. 看页面返回的具体错误
2. 看当前是不是最新代码
3. 看 Zeabur 是否已经重新构建镜像
4. 看挂载卷是否覆盖了文件
5. 再决定是改代码还是改环境

不要再直接凭感觉乱试。

## 10. 跟随上游升级建议（最小冲突路线）

目标：

- 保持系统骨架尽量不动
- 业务可用优先
- 主题/UI/SEO 逐步做成你想要的样子

建议的升级方式：

- 使用“上游仓库 + 合并更新”的方式，而不是覆盖代码。
- 每次升级后，重点检查上面第 9 节列出的文件改动是否被冲掉。

推荐流程：

1. 给官方仓库设置为上游（upstream），定期拉取更新
2. 上游有新版本后，把上游更新合并进当前 `main`
3. 如果出现冲突，优先保留：
   - Dockerfile / entrypoint / php.ini 等部署相关改动
   - 主题和前端展示改动
4. 合并完成后按第 5 节做一次回归验收，再 Redeploy

运营原则：

- 先把“最小可运营链路”稳定住（商品展示/下单/支付/发货）
- UI、SEO、合并货源规格化这些需求可以慢慢做，保证每一步都可回滚
