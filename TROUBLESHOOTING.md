# 故障排查记录

## 店铺共享：新商品全部接入失败

### 基本信息

- 发生时间：2026-07-23 至 2026-07-24
- 主站版本：acg-faka 3.5.5
- 远端协议：异次元 V3.1.2（重构后全新版）
- 相关修复：PR #1、PR #2

### 故障现象

- 店铺连接状态显示“正常”。
- 已经接入的远程商品可以同步成功。
- 新商品无论单个还是批量接入都失败。
- 前端只显示“拉取结束，总数量：N，成功：0，失败：N”，不显示具体原因。
- 商品管理中的商品总数没有增加。

这些现象说明远端店铺和密钥可以正常连接，问题发生在“读取商品详情后，将新商品保存到本地”这一步。

### 如何查看真实错误

3.5.5 的原始代码会吞掉单个商品的异常，只累计失败数量。PR #1 增加了远端商品导入日志。

进入应用容器，在网站根目录执行：

```bash
find runtime/log -type f -name 'shared_store_import_*.log' -exec tail -n 50 {} \;
```

日志文件实际位于：

```text
runtime/log/<数据库密码的 MD5>/shared_store_import_<货源ID>.log
```

不要公开日志目录名、数据库配置、商户密钥、Token、Cookie、支付私钥或用户隐私数据。

### 本次真实错误

```text
App\Service\Bind\Shared::AdjustmentPrice():
Argument #2 ($price) must be of type string, float given
```

报错位置为 `app/Controller/Admin/Api/Store.php` 中调用 `AdjustmentPrice()` 的代码。

### 根本原因

3.5.5 的远端商品校验会把 `price` 和 `user_price` 规范化为浮点数；但 `AdjustmentPrice()` 的参数类型仍声明为字符串。在严格类型模式下，PHP 不会自动接受这个类型差异，因此每个商品都会在加价计算之前失败。

大白话：价格内容没有错，但新版把价格装进了“数字盒子”，旧加价函数只接受“文字盒子”，所以程序拒绝继续处理。

### 修复方法

调用 `AdjustmentPrice()` 前，将两个价格参数转换成字符串：

```php
$config = $this->shared->AdjustmentPrice(
    (string)$item['config'],
    (string)$item['price'],
    (string)$item['user_price'],
    $premiumType,
    $premium
);
```

PR #1 还对旧货源可能返回的 `null` 或空字符串整数值增加了默认值兼容，并记录逐商品失败原因。PR #2 修复了本次真正导致全部导入失败的价格类型不匹配。

### 标准排查顺序

1. 先用一个商品复现，避免批量制造异常数据。
2. 看前端提示，确认失败发生在哪个功能。
3. 看后台操作日志，记录准确时间和操作对象。
4. 看对应的服务器错误日志，优先相信异常类型、文件和行号。
5. 根据日志检查代码，不要先删除店铺、商品或密钥。
6. 修改后先做最小范围测试，再批量操作。
7. 保留日志记录；若仍失败，根据新的第一条异常继续排查。

### 升级注意事项

- 升级前备份数据库和网站程序。
- 官方发布新版本时，先检查店铺共享、远程商品、价格和库存相关改动。
- 升级后先测试一个新商品接入和一个已有商品同步。
- 若官方文件覆盖本地补丁，应先比较对应代码，再决定是否保留补丁。
- 不要因为前端只显示“失败”就盲目回滚、删除数据或更换密钥。
